## 시작하며

커리어 전환은 단순한 직업 변경이 아닙니다. 기존 경험의 **재해석**이자, 새로운 맥락에서의 **가치 발견** 과정입니다.

저는 9년간 건설CM(Construction Management) 분야에서 일하며 복잡한 이해관계자 조율, 공정 관리, 리스크 분석을 수행했습니다. 이 경험이 AI 시스템 아키텍처 설계에 어떻게 연결되는지 이야기하려 합니다.

## 건설 현장에서 배운 것

건설 프로젝트는 본질적으로 **복잡 적응 시스템**입니다:

- **다수의 이해관계자**: 발주처, 설계사, 시공사, 감리단이 각자 다른 목표와 제약 조건을 가집니다.
- **불확실성 관리**: 날씨, 자재 수급, 인력 변동 등 예측 불가능한 변수가 상존합니다.
- **문서 기반 의사결정**: 수천 건의 도면, 시방서, 변경 관리 문서가 프로젝트의 "진실의 원천"입니다.

이런 환경에서 저는 자연스럽게 **시스템적 사고**를 체득했습니다.

## 전환의 계기

> "기존 도구로는 더 이상 현장의 복잡성을 감당할 수 없다."

이 깨달음이 전환의 시작이었습니다. 엑셀과 수기 관리로는 5개 이상의 현장을 동시에 운영하는 것이 물리적으로 불가능했고, 이 문제를 해결하기 위해 직접 시스템을 설계하기 시작했습니다.

### 첫 번째 시도: 업무관리 시스템

Python과 데이터베이스를 독학하며 만든 첫 시스템은 단순했지만 효과적이었습니다:

```python
# 초기 프로토타입의 핵심 구조
class ProjectTracker:
    def __init__(self, sites: list[str]):
        self.sites = sites
        self.issues = {}

    def report_issue(self, site: str, category: str, description: str):
        """현장별 이슈를 체계적으로 추적"""
        if site not in self.issues:
            self.issues[site] = []
        self.issues[site].append({
            "category": category,
            "description": description,
            "status": "open"
        })
```

## Transferable Skills

건설CM에서 AI 시스템 설계로의 전환에서 가장 가치 있었던 역량들:

1. **이해관계자 조율 → 요구사항 정의**: 서로 다른 목표를 가진 당사자들의 니즈를 구조화하는 능력
2. **공정 관리 → 워크플로우 설계**: 순서와 의존성을 파악하고 병렬화하는 사고방식
3. **리스크 관리 → 시스템 안정성**: "무엇이 잘못될 수 있는가"를 항상 먼저 생각하는 습관
4. **문서 관리 → 데이터 아키텍처**: 대량의 구조화된 정보를 관리하고 검색 가능하게 만드는 경험

## 앞으로의 방향

도메인 전환은 끝이 아니라 **시작**입니다. 건설 현장의 실제 문제를 AI 협업 시스템으로 번역할 수 있는 사람 — 이것이 제가 추구하는 포지셔닝입니다.

다음 글에서는 구체적인 프로젝트 사례를 통해 이 접근 방식이 어떤 결과를 만들어냈는지 공유하겠습니다.
