## Problem

건설사업관리(CM) 기술제안서는 한 건당 20여 종의 보고서와 슬라이드가 필요하고, 앞부분의 약속과 뒷부분의 설명이 충돌하지 않도록 전체가 하나의 스토리로 읽혀야 합니다. 기존 작성 방식에는 다음과 같은 병목이 있었습니다.

- **맥락 단절**: 여러 작성자가 분업하면서 문서 간 논리적 일관성이 깨지는 문제가 반복되었습니다.
- **반복 구조화 작업**: 입찰공고문, RFP, 과업내용서에서 핵심 정보를 추출하고 정리하는 초기 작업이 매번 수작업이었습니다.
- **자산 관리 부재**: 회사 공통 자료(실적, 가이드라인)와 사업별 자료가 섞여 재사용이 어려웠습니다.
- **품질 편차**: 작성자의 숙련도에 따라 산출물 품질이 크게 달라졌습니다.

## Role

2인 팀에서 시스템 전체 기획과 에이전트 개발을 담당했습니다.

- **자산 분류 체계 설계**: 회사 공통 자료(`companyinfo`), 작성 기준(`guideline`), 사업별 자료(`projectdata`) 3종으로 분리하는 구조를 정의했습니다.
- **다중 에이전트 워크플로우 설계**: 자산 분석 → 프로젝트 정보 추출 → 스냅샷 확정 → 계획 수립 → 실행의 단계별 파이프라인을 설계하고 구현했습니다.
- **오케스트레이션 엔진 개발**: Run 단위로 작업을 묶고, Phase → Stage → Task 계층에서 상태 추적·복구·재시도를 처리하는 실행 엔진을 구축했습니다.
- **프롬프트 엔지니어링**: 복잡도별 모델 선택(flash-lite/flash/pro)과 Pydantic 기반 응답 검증 체계를 설계했습니다.
- **산출물 체계 구현**: Markdown 보고서와 HTML 슬라이드를 1 Task = 1 Page 원칙으로 생성하는 결과물 파이프라인을 만들었습니다.

## System

| 구분 | 기술 |
|------|------|
| **언어** | Python |
| **프레임워크** | FastAPI |
| **AI/LLM** | Google Gemini API (flash-lite / flash / pro) |
| **데이터베이스** | Supabase (PostgreSQL), 15개 테이블 |
| **자동화** | 이벤트 기반 오케스트레이션 + 폴링 백업 |

### 기술적 도전과 해결

- **문서 간 일관성 유지**: 앞 단계의 판단(프로젝트 분석, 자산 요약)이 다음 단계로 누적되도록 스냅샷 기반 컨텍스트 전달 구조를 설계했습니다. 한 번의 Run 실행으로 전체 산출물이 연결된 스토리를 유지합니다.
- **대용량 문서 처리**: PDF 분할(45MB 기준), 이미지 변환, 멀티모달 분석을 결합해 수백 페이지 분량의 입찰 자료를 안정적으로 처리하는 파이프라인을 구축했습니다.
- **장애 복구**: Run 실행 중 LLM API 장애나 타임아웃이 발생하면 자동 복구(최대 3회 재시도)와 강제 복구 경로를 제공해 운영 안정성을 확보했습니다.

## Outcome

- **26개 API 엔드포인트, 16개 서비스**로 구성된 문서 생성 운영 체계를 구현했습니다.
- **20여 종의 보고서·슬라이드**를 하나의 실행으로 자동 생성하는 워크플로우를 구축했습니다.
- **3종 자산 타입** 분리로 회사 공통 자료의 안전한 재사용 체계를 만들었습니다.
- **15개 DB 테이블** 기반의 데이터 모델로 프로젝트·자산·실행 이력을 체계적으로 관리합니다.
- 이벤트 기반 오케스트레이션과 폴링 백업의 **이중화 실행 구조**로 운영 안정성을 확보했습니다.

### 핵심 메트릭

| 항목 | 수치 |
|------|------|
| API 엔드포인트 | 26개 |
| 서비스 모듈 | 16개 |
| 자동 생성 산출물 유형 | 20여 종 |

| 팀 구성 | 2인 (기획·개발 담당) |

### 이 프로젝트가 보여주는 것

이 프로젝트는 AI를 단순 텍스트 생성 도구가 아니라, 도메인 맥락을 이해하고 다단계 판단을 누적하는 **워크플로우 엔진**으로 설계할 수 있음을 검증한 사례입니다. 문서를 빨리 만드는 것이 아니라, 20여 종의 산출물이 하나의 스토리로 읽히도록 맥락을 설계하는 것이 핵심 도전이었습니다.
