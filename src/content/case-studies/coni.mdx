## Problem

AI 에이전트와 협업하면서 반복적으로 부딪히는 구조적 한계가 있었습니다.

- **세션 휘발성**: 대화가 끊기면 작업 맥락이 사라져 매번 처음부터 다시 설명해야 했습니다.
- **단일 에이전트 한계**: 하나의 AI만 쓸 수 있어 복잡한 작업을 분업하거나 병렬로 처리할 수 없었습니다.
- **프로젝트 기억 부재**: 과거 커밋, 설계 판단, 코드 의존성 정보가 AI에게 전달되지 않아 동일한 실수가 반복되었습니다.
- **프로젝트별 맞춤 부재**: AI가 프로젝트의 기술 스택, 컨벤션, 도구 환경을 모른 채 일반적인 답변만 제공했습니다.

## Role

개인 프로젝트로 기획, 아키텍처 설계, 전체 개발을 담당했습니다.

- **작업 추적 파이프라인 설계**: Objective → Workplan → Commit 3계층 구조를 정의하고, idle → breakdown → execute → verify → complete 5단계 상태 머신을 구현했습니다.
- **멀티 에이전트 오케스트레이션**: tmux 기반으로 Claude, Gemini, Codex 등 여러 CLI 에이전트를 동시 실행하고, 역할(architect, critic, worker)별 프롬프트 라우팅을 구축했습니다.
- **프로젝트 메모리 시스템**: 임베딩 + 리랭커 기반 시맨틱 검색, 코드 그래프, 영향 분석 기능을 구현해 프로젝트가 스스로를 기억하는 구조를 만들었습니다.
- **하네스 엔지니어링**: 7단계 자동 분석(스택 → 도메인 → 컨벤션 → 도구 → 가이드 → 설치 → 스킬 적용)으로 어떤 프로젝트에든 CONI를 맞출 수 있는 체계를 설계했습니다.
- **DB 이중화**: SQLite(로컬)와 Supabase(원격) 두 백엔드를 지원해, 개인 사용부터 팀 공유까지 유연하게 대응합니다.

## System

| 구분 | 기술 |
|------|------|
| **언어** | TypeScript |
| **런타임** | Node.js |
| **데이터베이스** | SQLite (로컬) + Supabase (원격) |
| **오케스트레이션** | tmux 세션 기반 멀티 에이전트 |
| **메모리** | OpenAI / Ollama 임베딩 + 리랭커 (하이브리드 검색) |
| **CLI 프레임워크** | CAC |
| **VCS 연동** | Git Hook 기반 커밋 자동 기록 |

### 기술적 도전과 해결

- **세션 연속성**: 모든 작업 상태를 DB에 기록하고, `coni status`가 현재 phase를 감지해 `next_steps`를 안내합니다. 새 세션에서 "작업 계속해줘" 한마디면 마지막 커밋 기반으로 정확히 이어갑니다.
- **에이전트 비종속성**: 특정 AI에 종속되지 않도록 에이전트 추상화 계층을 설계했습니다. Claude, Gemini, Codex 또는 향후 어떤 CLI 에이전트든 동일한 인터페이스로 연결됩니다.
- **비정상 상태 감지**: 6가지 비정상 상태(고아 Workplan, 미연결 커밋, 상태 드리프트 등)를 자동 감지하고 `coni reconcile`로 복구하는 체계를 구현했습니다.

## Outcome

- **대화 세션이 끊겨도 작업이 이어지는** 영속적 협업 흐름을 구축했습니다.
- **5단계 작업 파이프라인**(idle → breakdown → execute → verify → complete)으로 개발 워크플로우를 체계화했습니다.
- **7단계 하네스 엔지니어링**으로 어떤 프로젝트에든 맞춤 적용 가능한 구조를 만들었습니다.
- **6가지 비정상 상태 자동 감지**와 복구로 DB-Git 간 일관성을 보장합니다.
- **DB 이중화**(SQLite + Supabase)로 개인 사용과 팀 공유를 모두 지원합니다.
- 임베딩 기반 **시맨틱 검색과 코드 그래프**로 프로젝트 맥락을 기계가 이해할 수 있는 형태로 축적합니다.

### 핵심 메트릭

| 항목 | 수치 |
|------|------|
| 작업 파이프라인 단계 | 5단계 |
| 하네스 분석 프로세스 | 7단계 |
| 비정상 상태 감지 유형 | 6가지 |
| DB 백엔드 | 2종 (SQLite + Supabase) |
| 팀 구성 | 1인 (기획·설계·개발) |

### 이 프로젝트가 보여주는 것

이 프로젝트는 AI 협업의 핵심 병목이 모델 성능이 아니라 **맥락의 연속성과 역할 분리**에 있다는 가설을 검증한 사례입니다. 작업계획, 진행 상태, 커밋을 연결해 세션이 바뀌어도 작업이 이어지는 구조를 직접 설계·구현했으며, 이 경험은 반복 가능한 AI 협업 흐름이 실무 생산성의 기반이 된다는 확신으로 이어졌습니다.
