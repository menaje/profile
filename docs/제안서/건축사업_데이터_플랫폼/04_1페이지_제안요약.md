# 건축사업 데이터 플랫폼 1페이지 제안요약

## 한 줄 정의

건축사업의 설계, CM, 입찰, 실행, 유지관리 전 과정을 하나의 프로젝트 데이터 체계로 연결하고, 각 부서는 역할 기반 모듈로 일하며, AI는 권한과 보안을 전제로 필요한 정보만 제공하는 플랫폼

## 해결하려는 문제

건축사업은 하나의 프로젝트임에도 실제 운영에서는 설계사업, CM사업, 입찰, 실행, 유지관리 등 단계와 부서별로 분리 관리된다. 이 과정에서 데이터는 다음과 같은 문제를 겪는다.

- 부서가 바뀔 때 데이터와 맥락이 끊긴다.
- 기존 PMIS는 대체로 특정 부서 중심이라 다음 단계와 자연스럽게 연결되지 않는다.
- 인수인계는 문서 전달 중심이라 의사결정 배경과 변경 이력이 함께 넘어가지 않는다.
- 보안 때문에 전체 데이터를 폭넓게 공유하기 어렵다.

결국 같은 프로젝트를 각 부서가 따로 관리하면서, 프로젝트 전체 맥락은 시스템이 아니라 사람의 기억과 경험에 의존하게 된다.

## 제안하는 해결 방식

이 서비스는 모든 부서가 동일한 화면을 쓰게 하려는 것이 아니다. 대신 모든 부서가 동일한 프로젝트 데이터 체계를 공유하게 만든다.

구조는 세 가지 축으로 구성된다.

- 공통 프로젝트 데이터 체계
- 역할 기반 업무 모듈
- 보안 통제형 AI 정보 접근 계층

공통 데이터 체계는 프로젝트, 문서, 도면, 이슈, 변경, 승인, 일정, 원가, 리스크, 유지관리 자산 등 핵심 객체를 하나의 기준으로 연결한다. 각 부서는 자신에게 맞는 모듈과 화면을 사용하되, 그 아래의 데이터는 같은 구조 위에 존재한다.

AI는 이 데이터 구조를 바탕으로 사용자의 권한 범위 안에서 필요한 정보만 탐색하고 요약하며, 인수인계 브리핑, 변경 영향 분석, 누락 데이터 탐지 같은 기능을 제공한다.

## 왜 차별적인가

기존 PMIS가 특정 부서의 업무 효율화에 머무는 경우가 많다면, 이 플랫폼은 프로젝트 전 생애주기의 데이터 연속성을 핵심 가치로 둔다.

차별점은 다음과 같다.

- 부서별 업무관리 도구가 아니라 프로젝트 중심 데이터 운영 플랫폼이다.
- 역할별 모듈을 허용해 현업 수용성을 높인다.
- 데이터 연결 구조를 제품의 핵심으로 삼는다.
- AI를 단순 자동화가 아니라 권한 기반 정보 접근 인터페이스로 사용한다.
- 보안과 활용성을 동시에 고려한다.

## 기대 효과

- 부서 간 인수인계 품질 향상
- 의사결정 배경과 변경 이력의 축적
- 프로젝트 전반의 추적성과 책임성 강화
- 데이터 검색과 해석 시간 단축
- 향후 표준화, 분석, 자동화, 예측 기능으로의 확장 가능성 확보

## 이 서비스의 본질

이 아이디어의 본질은 새로운 PMIS를 하나 더 만드는 것이 아니다. 핵심은 건축사업을 "부서별 관리"가 아니라 "프로젝트 중심 데이터 운영"으로 재구성하는 것이다.

즉, 이 플랫폼은 다음 질문에 대한 답이다.

> 같은 프로젝트인데 왜 부서가 바뀔 때마다 다시 정리하고 다시 해석해야 하는가

이 서비스는 그 질문에 대해 다음과 같이 답한다.

> 프로젝트는 하나의 데이터 맥락으로 이어지고, 각 부서는 그 위에서 자신의 역할에 맞게 일해야 한다.
