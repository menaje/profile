# 건축사업 데이터 플랫폼 대외 설명 피치 요약

## 1. 한 줄 설명

건축사업의 설계, CM, 입찰, 실행, 유지관리 전 과정을 하나의 프로젝트 데이터 체계로 연결하고, AI가 권한 기반으로 필요한 정보만 제공하는 프로젝트 운영 플랫폼

## 2. 왜 이 제안이 필요한가

건축사업은 하나의 프로젝트지만, 실제 운영은 설계사업, CM사업, 입찰, 실행, 유지관리 등 부서와 단계별로 나뉘어 있다. 그 결과 같은 프로젝트가 다음 단계로 넘어갈 때 데이터와 맥락이 끊긴다.

실제 현장에서는 이런 문제가 반복된다.

- 인수인계가 사람과 문서 전달에 의존한다.
- 기존 PMIS는 특정 부서 최적화에 머문다.
- 변경 사유와 승인 이력이 함께 이어지지 않는다.
- 필요한 정보를 찾기 위해 실무자가 원문을 반복해서 뒤져야 한다.
- 보안 문제 때문에 전체 데이터를 넓게 공유하기 어렵다.

## 3. 무엇을 제안하는가

이 제안은 모든 부서가 같은 화면을 쓰게 하자는 것이 아니다. 핵심은 모든 부서가 같은 프로젝트 데이터 체계를 공유하도록 만드는 것이다.

구조는 세 가지로 정리된다.

- 공통 프로젝트 코어
- 역할 기반 업무 모듈
- 보안 통제형 AI 정보 접근 계층

즉, 통합 PMIS가 아니라 프로젝트 데이터 백본 위에 부서별 운영 도구를 얹는 구조다.

## 4. 무엇이 다른가

- 기존 PMIS는 부서 중심, 본 제안은 프로젝트 데이터 중심
- 기존 문서관리는 저장 중심, 본 제안은 관계와 맥락 중심
- 기존 AI 활용은 생성 중심, 본 제안은 권한 기반 접근 중심
- 기존 인수인계는 사람 중심, 본 제안은 구조화된 데이터 연속성 중심

## 5. 어떻게 시작하는가

이 제안은 처음부터 전사 통합으로 시작하지 않는다. 가장 큰 고통을 만드는 인수인계 구간 하나를 먼저 해결한다.

초기 적용 구간은 다음 둘 중 하나가 적합하다.

- 설계사업 -> CM사업 전환
- 입찰 -> 실행 전환

이 구간에서 문서, 이슈, 변경, 회의, 승인 이력을 연결하고, AI 기반 인수인계 브리핑을 제공하면 첫 가치를 검증할 수 있다.

## 6. 기대되는 변화

- 부서 간 인수인계 품질 향상
- 프로젝트의 의사결정 배경과 변경 이력 축적
- 실무자의 정보 탐색 시간 단축
- 보안과 활용성을 동시에 고려한 데이터 운영 체계 확보
- 향후 다른 부서와 프로젝트로 확장 가능한 기반 확보

## 7. 왜 이 문제가 풀릴 수 있는가

이 문제는 현장을 직접 경험한 사람 아니면 정확히 정의하기 어렵다. 제안자는 건축사사무소와 CM 환경에서 프로젝트 운영, 공무, 협업, 데이터 단절의 문제를 실제로 경험했고, 이를 해결하기 위한 IT와 AI 기반 시스템을 직접 설계하고 적용해 본 배경을 갖고 있다.

즉, 이 제안은 추상적인 디지털 전환 구호가 아니라, 현장 문제를 구조적으로 다시 설계하려는 시도다.

## 8. 한 문장 요약

> 각 부서는 자신의 방식으로 일하되, 프로젝트는 하나의 데이터 맥락 안에서 운영되게 한다.
