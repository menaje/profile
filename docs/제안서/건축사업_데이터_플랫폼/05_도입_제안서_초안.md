# 건축사업 데이터 플랫폼 도입 제안서 초안

## 1. 제안 목적

본 문서는 건축사사무소 또는 CM 조직이 동일 프로젝트를 여러 사업 영역과 부서에 걸쳐 수행하는 과정에서 발생하는 데이터 단절 문제를 해결하기 위한 서비스 제안서다.

제안의 핵심은 새로운 업무 툴 하나를 더 도입하자는 것이 아니다. 더 중요한 목표는 귀사의 설계사업, CM사업, 입찰, 실행, 유지관리 과정이 서로 다른 방식으로 운영되더라도, 프로젝트의 데이터와 맥락은 하나의 체계 안에서 끊기지 않도록 만드는 것이다.

## 2. 현재 운영 방식의 한계

건축사업은 본질적으로 하나의 프로젝트이지만, 실제 운영에서는 사업 영역과 부서에 따라 나뉘어 관리된다. 이 구조는 역할 분담에는 유리하지만 다음과 같은 문제를 만든다.

- 같은 프로젝트가 단계나 부서가 바뀔 때마다 다시 정리되고 다시 해석된다.
- 기존 PMIS는 특정 부서 중심으로 운영되어 다음 단계와 자연스럽게 이어지지 않는다.
- 인수인계가 문서 전달 중심으로 진행되어 의사결정 배경, 변경 사유, 승인 흐름이 누락되기 쉽다.
- 실무자는 필요한 정보를 찾기 위해 원문을 반복해서 탐색하고 맥락을 재구성해야 한다.
- 보안과 권한 문제로 인해 전체 데이터를 자유롭게 공유하는 방식은 현실적으로 어렵다.

결국 프로젝트 전체의 흐름은 시스템이 아니라 사람의 기억과 경험에 의존하게 된다.

## 3. 제안하는 방향

제안하는 방향은 모든 부서에 동일한 화면과 동일한 방식의 업무를 강제하는 것이 아니다. 대신 모든 부서가 동일한 프로젝트 데이터 체계를 공유하도록 만드는 것이다.

즉, 각 부서는 자신의 역할에 맞는 화면과 프로세스를 사용하되, 프로젝트를 구성하는 핵심 데이터는 공통 구조 안에서 연결되고 누적되도록 설계한다.

이 방식은 다음 질문에 대한 답이다.

> 같은 프로젝트인데 왜 부서가 바뀔 때마다 다시 정리하고 다시 해석해야 하는가

우리의 제안은 다음과 같다.

> 부서는 다르게 일할 수 있어도, 프로젝트 데이터는 하나의 맥락 안에서 이어져야 한다.

## 4. 제안 솔루션 구조

서비스 구조는 세 계층으로 설명할 수 있다.

### 4.1 공통 프로젝트 코어

프로젝트, 문서, 도면, 이슈, 변경, 승인, 일정, 리스크, 자산 등 핵심 객체를 하나의 기준으로 관리한다. 이 계층이 데이터 연속성의 중심이 된다.

### 4.2 역할 기반 업무 모듈

설계, CM, 입찰, 실행, 유지관리 조직은 각자의 화면과 프로세스로 업무를 수행한다. 즉, 통합 화면이 아니라 역할 기반 운영 구조를 지향한다.

### 4.3 보안 통제형 AI 접근 계층

사용자는 자신의 권한 범위 안에서 필요한 정보만 AI를 통해 탐색하고 브리핑 받을 수 있다. AI는 원문을 대체하는 것이 아니라, 근거와 출처를 기반으로 정보를 해석하고 전달하는 인터페이스로 동작한다.

## 5. 귀사에 기대되는 효과

### 5.1 조직 관점

- 프로젝트 전 단계의 추적성과 투명성 확보
- 부서 간 인수인계 누락 감소
- 의사결정과 변경 이력의 장기 축적

### 5.2 부서장 관점

- 설계, CM, 입찰, 실행 간 전달 품질 향상
- 프로젝트 상태와 리스크를 빠르게 파악 가능
- 책임과 근거 추적이 쉬운 구조 확보

### 5.3 실무자 관점

- 필요한 자료 탐색 시간 감소
- 관련 문서, 이슈, 회의, 승인 이력을 한 맥락에서 확인 가능
- AI를 통한 빠른 질의응답과 인수인계 브리핑 활용 가능

## 6. 권장 도입 방식

본 제안은 처음부터 전사 통합 플랫폼을 한 번에 구축하는 접근을 권장하지 않는다. 가장 적합한 방식은 하나의 전환 구간을 파일럿으로 정해 가치를 먼저 검증하는 것이다.

권장 도입 순서는 다음과 같다.

1. 귀사 내에서 데이터 단절이 가장 심한 전환 구간을 선정한다.
2. 해당 구간의 공통 데이터 객체와 인수인계 기준을 정의한다.
3. 문서, 이슈, 변경, 회의, 승인 이력을 연결하는 파일럿을 운영한다.
4. AI 기반 브리핑과 누락 탐지 기능으로 실무 활용성을 검증한다.
5. 검증된 후 인접 부서와 단계로 확장한다.

## 7. 초기 적용 범위 제안

초기 적용 구간은 다음 두 가지가 유력하다.

- 설계사업 -> CM사업 전환
- 입찰 -> 실행 전환

이 두 구간은 데이터 단절, 인수인계 누락, 책임과 근거 추적의 문제가 가장 선명하게 드러나는 구간이며, 플랫폼의 가치도 빠르게 확인할 수 있다.

## 8. 제안의 차별점

이 제안의 차별점은 단순히 기능을 더 많이 제공하는 데 있지 않다.

- 기존 PMIS가 부서 중심이라면, 본 제안은 프로젝트 데이터 중심이다.
- 기존 문서관리가 저장 중심이라면, 본 제안은 관계와 맥락 중심이다.
- 기존 AI 활용이 생성 중심이라면, 본 제안은 권한 통제형 정보 접근 중심이다.
- 기존 인수인계가 사람 중심이라면, 본 제안은 구조화된 데이터 연속성 중심이다.

## 9. 협업 전제

이 제안은 기존 시스템을 즉시 전면 대체하는 방식보다, 우선 기존 운영 위에 데이터 연속성을 부여하는 방식이 현실적이다. 따라서 도입 초기에는 다음 전제가 중요하다.

- 기존 문서 체계와 프로젝트 운영 흐름을 존중한다.
- 공통 데이터 객체를 최소 범위부터 정의한다.
- 권한과 보안을 시스템 설계의 출발점으로 둔다.
- AI 응답은 항상 출처와 근거를 함께 제공한다.

## 10. 결론

본 제안의 본질은 새 PMIS 하나를 만드는 데 있지 않다. 더 중요한 것은 귀사의 프로젝트 운영 방식을 "부서별 관리"에서 "프로젝트 중심 데이터 운영"으로 한 단계 전환하는 것이다.

이 플랫폼은 각 부서가 자신의 방식으로 일하더라도, 프로젝트 자체는 하나의 데이터 맥락 안에서 이어지도록 만든다. 그 점에서 본 제안은 기능 추가가 아니라 운영 구조의 개선 제안으로 이해하는 것이 가장 적절하다.
