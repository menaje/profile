## 핵심 프로젝트

### 1. CONI | 개인 프로젝트

| 항목 | 내용 |
|------|------|
| **기간** | 2025년 11월 시작, 2026년 3월 20일 기준 진행 중 |
| **유형** | 개인 프로젝트 |
| **역할** | 기획, 아키텍처 설계, 전체 개발 |
| **한 줄 요약** | Objective → Workplan → Commit 구조로 AI 협업을 추적하고, 대화 세션이 바뀌어도 작업이 이어지도록 설계한 CLI 기반 협업 시스템 |

#### Problem

- 단일 AI 에이전트 중심 작업은 전략 수립, 실행, 검증이 한 대화창에 뒤섞여 깊이와 정확도를 동시에 확보하기 어려웠습니다.
- 대화 세션이 종료되면 작업 상태와 의사결정 맥락이 사라져, 다음 세션에서 같은 설명을 반복해야 했습니다.
- 여러 AI CLI를 병렬로 활용하고 싶어도 역할 분리, 프롬프트 라우팅, 결과 회수가 체계화돼 있지 않았습니다.
- 프로젝트가 시간이 지나도 스스로를 기억하도록 만들기 위해 코드, 문서, 커밋을 연결하는 메모리 계층이 필요했습니다.

#### Role

- Objective, Workplan, Commit을 하나의 작업 계층으로 묶는 협업 구조를 설계했습니다.
- tmux 기반 멀티 에이전트 오케스트레이션 흐름과 역할별 프롬프트 전달 방식을 정의했습니다.
- 새 세션에서도 최근 커밋과 진행 중 Workplan을 기준으로 작업을 이어갈 수 있는 상태 추적 구조를 만들었습니다.
- 하네스 엔지니어링을 통해 프로젝트별 스택, 컨벤션, 도구 상태를 분석하고 에이전트 스킬에 주입하는 흐름을 설계했습니다.
- 코드, 문서, 커밋을 시맨틱하게 연결하는 프로젝트 메모리 방향을 설계해 장기적인 컨텍스트 유지 문제를 풀고자 했습니다.

#### Stack

| 구분 | 기술 |
|------|------|
| **언어/런타임** | TypeScript, Node.js |
| **오케스트레이션** | tmux, 역할 기반 CLI 에이전트 협업 |
| **상태 저장** | Supabase (PostgreSQL), SQLite |
| **작업 추적** | Git 연동, Commit 기반 상태 연결 |
| **확장 축** | 임베딩 기반 메모리, 리랭커, 하네스 엔지니어링 |

#### Outcome

- **대화가 끊겨도 작업이 이어지는 협업 구조**를 구현했습니다.
- **여러 AI CLI를 병렬로 운영하는 오케스트레이션 체계**를 정리해, 전략/실행/검증 역할을 분리할 수 있게 했습니다.
- **프로젝트 메모리 방향성**을 명확히 정의해 코드, 문서, 커밋을 연결하는 장기 맥락 관리 기반을 만들었습니다.
- **프로젝트 맞춤 하네스 워크플로우**를 통해 새 저장소에 들어가도 동일한 협업 프로토콜을 빠르게 적용할 수 있게 했습니다.

### 2. 제안서 에이전트 | (주)희림종합건축사사무소

| 항목 | 내용 |
|------|------|
| **기간** | 2024년 5월 시작, 2026년 3월 20일 기준 진행 중 |
| **유형** | 회사 프로젝트 |
| **역할** | 기획 및 에이전트 개발 |
| **팀 구성** | 2인 |
| **한 줄 요약** | 20여 종의 보고서와 슬라이드가 한 흐름 안에서 생성되도록 다중 에이전트 오케스트레이션과 컨텍스트 전달 체계를 설계한 문서 자동화 시스템 |

#### Problem

- 제안서 작성은 산출물 수가 많아질수록 앞뒤 내용 불일치와 톤 분산이 자주 발생했습니다.
- 여러 보고서와 슬라이드가 동시에 만들어지다 보니, 전체 스토리라인을 유지하면서 병렬 생성하는 구조가 필요했습니다.
- 외부 LLM API 호출 실패, 네트워크 오류, 중간 단계 중단 같은 운영 리스크가 누적됐습니다.
- 단순 프롬프트 확장이 아니라 단계별 결과를 축적해 다음 단계 품질을 올리는 설계가 필요했습니다.

#### Role

- Phase → Stage → Task로 이어지는 계층형 작업 분할 구조를 설계했습니다.
- 앞 단계 결과를 다음 단계로 누적 전달하는 컨텍스트 엔지니어링 방식을 정리했습니다.
- 다중 에이전트 오케스트레이션 구조를 적용해 역할별 생성과 검증을 분리했습니다.
- 운영 환경을 고려해 재시도, 복구, 헬스체크를 포함한 안정성 설계를 구현했습니다.
- 26개 API 엔드포인트, 16개 서비스로 구성된 실행 체계를 정리해 운영 가능한 수준으로 고도화했습니다.

#### Stack

| 구분 | 기술 |
|------|------|
| **언어** | Python |
| **프레임워크** | FastAPI |
| **LLM** | Google Gemini API |
| **상태 관리** | Supabase (PostgreSQL) |
| **아키텍처** | 계층형 + 이벤트 기반 하이브리드 |

#### Outcome

- **한 번의 실행으로 20여 종의 보고서와 슬라이드 자동 생성** 체계를 구축했습니다.
- **컨텍스트 누적 전달 방식**으로 문서 간 용어와 스토리 흐름의 일관성을 높였습니다.
- **재시도, RecoveryService, Health Check**를 포함한 운영 안정성을 확보했습니다.
- 문제 해결 과정에서 정리한 방식이 이후 **Context Engineering**과 **Multi-Agent Orchestration**의 핵심 원리와 맞닿아 있음을 확인했습니다.

### 3. WPM | 개인 프로젝트

| 항목 | 내용 |
|------|------|
| **기간** | 2025년 11월 시작, 2026년 3월 20일 기준 진행 중 |
| **유형** | 개인 프로젝트 |
| **역할** | 기획 및 개발 |
| **한 줄 요약** | Git 커밋과 작업 추적을 하나의 흐름으로 묶고, AI 에이전트 협업 프로토콜을 운영 가능한 CLI로 만든 프로젝트 관리 도구 |

#### Problem

- AI 에이전트는 코드를 작성해도 작업 단위, 진행 상태, 커밋 이력이 분리돼 있으면 협업의 연속성이 약했습니다.
- 커밋이 실제 Workplan과 연결되지 않으면 무엇을 왜 바꿨는지 추적하기 어려웠습니다.
- 작업관리 도구 오류가 Git 커밋 자체를 막아버리면 실무 흐름이 멈추는 위험이 있었습니다.
- 사람과 여러 AI 에이전트가 함께 쓰는 인터페이스에는 명확한 프로토콜과 fail-safe 설계가 필요했습니다.

#### Role

- Objective, Workplan, Commit을 연결하는 CLI 기반 작업관리 모델을 설계했습니다.
- 커밋 메시지, Git Hook, DB 기록을 연결해 작업 이력을 자동으로 추적하는 흐름을 구현했습니다.
- 로비 세션, 작업 세션, 후속 세션을 구분해 세션 기반 작업 시간 측정 구조를 만들었습니다.
- 비정상 상태를 감지하고 다음 행동을 제안하는 Guardian 성격의 검증 로직을 설계했습니다.
- 도구가 실패해도 Git 커밋이 완전히 막히지 않도록 exit 0 기반 fail-safe 정책을 적용했습니다.

#### Stack

| 구분 | 기술 |
|------|------|
| **언어/런타임** | TypeScript, Node.js 18+ |
| **CLI** | CAC |
| **상태 저장** | Supabase (PostgreSQL) |
| **Git 연동** | simple-git, Git Hook |
| **확장 계획** | Vector DB 기반 시스템 맥락 관리 |

#### Outcome

- **Git 커밋 기반 작업 자동 추적** 체계를 구현했습니다.
- **AI 에이전트 협업 프로토콜**을 설계해 역할과 상태 연결 방식을 명확히 했습니다.
- **세션 기반 정확한 작업 시간 추적**과 **워크플로우 가이던스**를 구현했습니다.
- **6가지 비정상 상태 자동 감지**를 포함한 검증 체계를 마련했습니다.
- YAML frontmatter 기반 문서 메타데이터 검색 구조를 도입해 문서와 작업의 연결성을 높였습니다.

### 4. CM 업무관리 시스템 | (주)정림CM

| 항목 | 내용 |
|------|------|
| **기간** | 2023년 1월 ~ 2024년 5월 |
| **유형** | 회사 프로젝트 |
| **역할** | 기획 및 개발 |
| **적용 범위** | 대전 5개 현장 통합건설사업관리 |
| **한 줄 요약** | 5개 현장의 문서, 계약, 사업비 데이터를 한 흐름으로 연결하고 문서 등록·요약·보고를 자동화한 현장형 협업 시스템 |

#### Problem

- 문서 중심의 CM 업무는 정보가 분산돼 있어 필요한 자료를 찾고 맥락을 설명하는 데 시간이 많이 들었습니다.
- 5개 현장을 동시에 관리하는 상황에서 데이터 구조가 다르면 협업과 의사결정이 느려졌습니다.
- 비IT 사용자가 많아 복잡한 시스템은 도입 자체가 실패할 가능성이 컸습니다.
- 월간보고서 작성과 문서 등록 같은 반복 업무를 줄이지 않으면 시스템이 현장에 남기 어려웠습니다.

#### Role

- Obsidian 기반 맞춤형 문서관리 구조를 설계하고 현장 흐름에 맞게 적용했습니다.
- OCR, LLM, 문서 등록 자동화를 결합해 비정형 문서를 구조화하는 파이프라인을 만들었습니다.
- 사업비, 계약, 문서 메타데이터를 연결해 현장별 상황을 빠르게 파악할 수 있게 했습니다.
- 월간보고서 자동 생성 흐름을 구성해 실무자의 반복 작업을 줄였습니다.
- 도구를 배포하는 데서 끝내지 않고 현장 전체 협업 방식에 실제로 안착시키는 역할까지 담당했습니다.

#### Stack

| 구분 | 기술 |
|------|------|
| **플랫폼** | Obsidian |
| **언어** | Python |
| **AI** | OpenAI GPT API, Gemini API |
| **Vision/OCR** | Apple Vision, Gemini 멀티모달 |
| **자동화** | Python 스크립트, Markdown 기반 지식 관리 |

#### Outcome

- **5개 현장 통합 문서 중앙 관리** 체계를 구축했습니다.
- **OCR + LLM 기반 문서 자동 등록 및 요약** 흐름을 구현했습니다.
- **월간보고서 자동 생성 파이프라인**을 도입해 반복 업무를 줄였습니다.
- 결과적으로 **현장 전체의 핵심 협업 시스템으로 정착**시켜, 기술 도입과 현장 안착을 동시에 검증했습니다.
