CONI (구 WPM)

One Line Summary

Objective → Workplan → Commit 구조로 AI 협업과 Git 기반 작업 추적을 연결하고, 대화 세션이 바뀌어도 작업이 이어지도록 설계한 CLI 기반 협업 시스템

Problem

  • 단일 AI 에이전트 중심 작업은 전략 수립, 실행, 검증이 한 대화창에 뒤섞여 깊이와 정확도를 동시에 확보하기 어려웠습니다.
  • 대화 세션이 종료되면 작업 상태와 의사결정 맥락이 사라져, 다음 세션에서 같은 설명을 반복해야 했습니다.
  • 여러 AI CLI를 병렬로 활용하고 싶어도 역할 분리, 프롬프트 라우팅, 결과 회수가 체계화돼 있지 않았습니다.
  • Git 커밋과 Workplan 이력이 분리되면 무엇을 왜 바꿨는지 추적하기 어려웠고, 작업관리 도구 오류가 실무 흐름 전체를 막아서는 안 됐습니다.
  • 프로젝트가 시간이 지나도 스스로를 기억하도록 만들기 위해 코드, 문서, 커밋을 연결하는 메모리 계층이 필요했습니다.

Role

  • Objective, Workplan, Commit을 하나의 작업 계층으로 묶는 협업 구조를 설계했습니다.
  • tmux 기반 멀티 에이전트 오케스트레이션 흐름과 역할별 프롬프트 전달 방식을 정의했습니다.
  • 새 세션에서도 최근 커밋과 진행 중 Workplan을 기준으로 작업을 이어갈 수 있는 상태 추적 구조를 만들었습니다.
  • 커밋 메시지, Git Hook, DB 기록을 연결해 작업 이력을 자동으로 추적하는 흐름과 세션 기반 작업 시간 측정 구조를 구현했습니다.
  • 비정상 상태를 감지하고 다음 행동을 제안하는 검증 로직과 fail-safe 정책을 설계했습니다.
  • 하네스 엔지니어링을 통해 프로젝트별 스택, 컨벤션, 도구 상태를 분석하고 에이전트 스킬에 주입하는 흐름을 설계했습니다.

Outcome

  • 대화가 끊겨도 작업이 이어지는 협업 구조를 구현했습니다.
  • 여러 AI CLI를 병렬로 운영하는 오케스트레이션 체계를 정리해, 전략/실행/검증 역할을 분리할 수 있게 했습니다.
  • Git 커밋 기반 작업 자동 추적과 세션 기반 작업 시간 측정을 구현해 협업 이력의 연속성을 높였습니다.
  • 6가지 비정상 상태 자동 감지와 fail-safe 정책으로 도구 장애가 실무 흐름을 막지 않도록 했습니다.
  • 프로젝트 메모리 방향성을 명확히 정의해 코드, 문서, 커밋을 연결하는 장기 맥락 관리 기반을 만들었습니다.
  • 프로젝트 맞춤 하네스 워크플로우를 통해 새 저장소에 들어가도 동일한 협업 프로토콜을 빠르게 적용할 수 있게 했습니다.

  • 제안서 작성은 산출물 수가 많아질수록 앞뒤 내용 불일치와 톤 분산이 자주 발생했습니다.

  • 운영 환경을 고려해 재시도, 복구, 헬스체크를 포함한 안정성 설계를 구현했습니다.
  • 26개 API 엔드포인트, 16개 서비스로 구성된 실행 체계를 정리해 운영 가능한 수준으로 고도화했습니다.
